# HexForge Requirements

## 1. Functional Requirements

### 1.1 Ingestion & Validation
- **FR1**: The system must accept a valid Hugging Face URL pointing to a `.gguf` file or a repository containing a GGUF file.
- **FR1.1**: The system must query the destination Hugging Face repository before queuing a job to verify if the requested model for the specific SoC architecture (`{Original_Model_Name}-{SoC_Model}-QAIRT`) has already been successfully compiled. If cached, the job is instantly marked completed.
- **FR2**: The system must dynamically fetch the list of supported phone models and their mapped Snapdragon Chipsets (e.g., SM8550) and Hexagon DSP Architectures (e.g., v73) from a backend database, avoiding hardcoded lists in the frontend to enable easy updates.
- **FR3**: The backend and frontend must parse the GGUF billion-parameter size (including decimals, e.g., "3B", "1.5B") from the URL using robust float parsing to prevent validation bypasses.
- **FR4**: Both the frontend UI and the backend API must validate the model size against the device's maximum NPU capacity. Requests exceeding the limit must be rejected with an actionable error message informing the user of the billion parameter limit and recommending smaller models or different quantization levels.
- **FR4.1**: The backend must also explicitly validate the actual file size of the GGUF via a lightweight request before downloading. This prevents malicious "filename gaming". The threshold must include reasonable "wiggle room" to account for various quantization levels (e.g., Q8 vs Q4) for a valid parameter size, but still reliably block distinct architectural mismatches (e.g., an 80GB 70B model file sent to a device only expecting a max 3B footprint).

### 1.2 Orchestration
- **FR5**: The system must maintain a durable job queue (FIFO) utilizing Google Cloud Tasks for guaranteed exactly-once and in-order execution with exponential backoff. Task creation must use **idempotent deterministic names** derived from the `job_id` to prevent duplicate task dispatch on application-level retries.
- **FR5.1**: The system must provide real-time queue position feedback to the user (e.g., "Position 3 in queue") to set accurate expectations regarding wait times before their job begins provisioning.
- **FR5.2**: The backend compilation pipeline must operate independently of the frontend client. If a user disconnects or closes their browser, the queued or processing job must still proceed to completion on the backend.
- **FR6**: The system must be capable of dynamically sizing host VM RAM and allocating high-speed Local SSD Swap partitions based on the targeted DSP architecture's compilation needs.
- **FR7**: The system must utilize Google Cloud Batch to autonomously manage the compute lifecycle, ensuring workers scale to zero natively when the target execution completes or fails, avoiding manual idle state logic.

### 1.3 Compilation Pipeline
- **FR8**: The worker must download the GGUF model from Hugging Face natively and must leverage intermediate `.cpp` and `.bin` state extraction using `GenAIBuilderFactory` cache_root persisted across node preemptions.
- **FR9**: The worker must utilize the Qualcomm AI Runtime (QAIRT) `GenAIBuilderFactory` targeting the `HTP` backend. `compile.py` must forcefully pre-flight RAM and Swap limits (blocking OS OOM killers gracefully if $< 128 \text{GB}$).
- **FR10**: The worker must integrate advanced Snapdragon architectural constraints: `fp16_relaxed_precision=1` mixed-precision enablement, `-O3` (HTP_OPTIMIZE_FOR_INFERENCE_O3) for symmetrical L2 cache tiling, `--dump_lut` for embedding payload detachment, and `--lm_head_precision BF16` to preserve output projection dynamic range against transformer activation outliers. It must also ensure proper `Qnn_FloatBlockEncoding_t` representation for hierarchical GGUF K-Quants, prioritize **IQ4_NL** for sensitive transformer blocks, and enable **per-channel quantization** for outlier isolation. Context weight sharing must be enabled to mitigate out-of-memory errors for dynamically scaled generation geometries. The system must also support **L2 Cache Elevation**, precise **Byte-Granularity Allocation (v81+)**, and footprint-halving **8-Bit KV Cache Compression**.
- **FR11**: If the targeted DSP is `v73`, the system must aggressively leverage `fastrpc_repack.json` blockwise expansion. If parameters exceed 3 Billion, it must autonomously invoke `SplitModelConfig` to cleanly sever the monolithic architecture and bypass 32-bit memory limits per execution context.
- **FR22**: The compiler must enforce HVX 128-byte super-group alignment (`REPACK_FOR_HVX=1`) for all Q4-class quantizations to fully saturate the Hexagon Vector eXtension registers and prevent 7/8ths bandwidth waste.
- **FR23**: The compiler must enforce NHWC tensor layout (`--force_nhwc 1`) to prevent QNN SDK runtime Transpose op injection for models originating with NCHW (PyTorch) data ordering.
- **FR24**: The compiler must offload the logits/vocabulary projection tensor to CPU (`--lm_head_backend CPU`) for all architectures where `NeedsLogitsOffload=true` to prevent 32-bit address space overflow on large-vocabulary models.
- **FR25**: The compiler must enforce incremental mmap-budget (`--mmap_budget 25`, configurable per hardware profile) for all context binary loading to prevent contiguous allocation OOM.
- **FR26**: Multi-session graph chunking via `SplitModelConfig` must apply to **all** 32-bit cDSP architectures (v68, v69, v73, 698) whenever the estimated runtime memory footprint ($M_{est}$) exceeds `MaxSessionMemoryGB`.
- **FR27**: Support Grouped-Query Attention (GQA) and Multi-Query Attention (MQA) via native HTP broadcast logic and `num_kv_heads` injection.
- **FR28**: Enforce HMX head-dimension alignment (multiple of 64) via automated graph padding operations to prevent perplexity degradation.
- **FR29**: Implement automated legacy hardware upcasting from INT4 to Q8_0 for architectures without native INT4 acceleration.
- **FR30**: The compiler must implement **IQ4_NL + Q8_0 Hybrid Quantization** for native INT4 silicon, prioritizing transformer blocks while protecting sensitive layers.
- **FR31**: The compiler must support **Speculative Decoding (SSD-Q1)** branches to reclaim idle HMX hardware rows.
- **FR32**: The compiler must enforce **RoPE AoS Interleaving** for peak vector register efficiency.
- **FR33**: The compiler must enable **Multi-Session Shared Memory** (KV share + Q8_0_32 quantization) when splitting models.
- **FR34**: The system must export a **Standalone Deployment Bundle** including a Fast Tokenizer JSON and LoRA adapter integration.
- **FR35**: The compiler must support **Mixture of Experts (MoE)** via graph switching logic on capable NPUs.
- **FR36**: The compiler must implement **KV Cache spill-fill buffers** to handle memory paging for long context windows.
- **FR37**: The system must provide a **Custom Operator Fallback** skeleton via OpPackage generation. The generator must enforce **BHWC 4D layout** and `const` parameter constraints and ensure 128-byte HVX alignment to ensure universal GGUF support (e.g., HardSwish, SwiGLU). It must automatically configure `architecture.gating=gated` for SwiGLU-detected models.
- **FR38**: The compiler must support **LongRoPE (Rotary Scaling)** for extended context windows.
- **FR39**: The system must leverage **INT2 and FP8 Precision** on Hexagon v79 and v80 architectures.
- **FR40**: The compiler must strictly enforce Crouton layout and VTCM placement for intermediate registrations to maximize spatial locality. It must also inject `operation.normalization` and `operation.normalization_epsilon` parameters derived from GGUF metadata.
- **FR41**: The system must support representative calibration dataset ingestion to refine static quantization scaling factors.
- **FR42**: The compiler must implement **Profile Bucketing**, generating an array of binaries for sequence lengths `[128, 512, 1024, 2048, 4096]` to support runtime graph switching.
- **FR43**: The system must implement **FastRPC Fragmentation Mitigation** for 32-bit HTP sessions (v73, v75) to prevent linker address space exhaustion.
- **FR44**: The compiler must leverage **E4M3/E5M2 Mixed-Precision FP8** on v79/v80 architectures.
- **FR45**: The system must detect RoPE organization and trigger an **AOT AoS Interleaving pass** if the source GGUF uses SoA layout.
- **FR46**: The compiler must introduce a JIT compilation fallback to the LiteRT Unified API when the low-level QNN composer encounters structurally incompatible neural architectures.
- **FR47**: The compiler must inject a mandatory 128-byte trailing dimension padding pass for non-HMX auxiliary tensors to prevent HVX hardware memory-alignment crashes.
- **FR48**: The compiler must parse `architecture.connector` from GGUF metadata to dictate native sequential or parallel topology mapping.
- **FR49**: The compiler must derive the attention head dimension mathematically ($E_{total} / N_{heads}$) if it is omitted in the source metadata to prevent precision starvation.
- **FR50**: The compiler must support **SpinQuant** orthogonal rotations prior to weight quantization to eliminate numerical outliers.
- **FR51**: The compiler must provide **Sequential MSE (SMSE)** quantization schemas to iterately refine scale and offset encodings during the offline phase.
- **FR52**: The compiler must enforce **FP32 Mathematical Firebreaks** on volatile summation nodes to prevent cascading NaN failures during high-parameter inference.
- **FR53**: The compiler must support **DeepSeek MLA Unrolling** into dense Multi-Head Attention to maintain HTP operability on custom topologies.
- **FR54**: The compiler must detect **Sliding Window Attention (SWA)** constraints from Mistral architectures. For 32-bit v73/v75 devices, it must mandate enforcing circular KV buffers that match the sliding window, triggering context spilling if memory caps are compromised.
- **FR55**: The compiler must utilize a `QNNOptimizerEngine` to mutate the graph structure with targeted surgeries (e.g., `MatmulAddFusion` supporting explicit `MaskedSoftmax` & `MatMul_bias` logic replacements, `SimplifiedLayerNormToL2Norm`, `RemoveRopeMultiCache`, `AttentionMaskToSequenceLengths`, and `WeightRotation`) to streamline NPU evaluation and protect 4-bit fidelity.
- **FR55.1**: The system must actively generate vector-aligned mathematical tensor pads on standard trailing dimensions, intrinsically rounding width/height arrays to strict SIMD alignment requirements through `MemoryAlignment` pre-processing.
- **FR56**: The compiler must integrate the `tuner.optimize` API alongside strict mathematical `ceil(channels / 4.0)` Packed Channel boundary protections and SpaceToDepth injections to automatically determine the most efficient multi-dimensional tensor tiling boundaries within the restricted VTCM.
- **FR57**: The compiler must implement a W4A16 fallback execution schema (INT4 weights + FP16 Activations) via `fp16_kernels` injection on older DSPs to mitigate total Q8_0 memory bandwidth starvation.
- **FR58**: The compiler must incorporate Shared Context Memory capabilities using `QNN_HTP_MEM_SHARED_BUFFER` and strict file read budgeting mapped dynamically per hardware profile.
- **FR59**: The compiler must leverage a graph rewriter (`HTPCompilerGraphRewriter`) to explicitly format input shapes to 4D BHWC and reject asymmetric zero-points to comply with v79/v81 silicon logic restrictions.
- **FR60**: The compiler must programmatically generate Gen AI Composer arguments embedding 32-byte alignment barriers and dynamic context limits matching runtime session capacities boundaries.
## 2. Performance Requirements

- **PR1**: The compilation pipeline must target peak silicon utilization:
  - **v79**: 45 TOPS (INT8)
  - **v75**: 34 TOPS (INT8)
  - **v73**: 26 TOPS (INT8)
- **PR2**: The system must maintain perplexity retention by enforcing HMX block alignment (64-element multiples) and avoid software dequantization on legacy hardware by upcasting to Q8_0.

### 1.4 Egress
- **FR12**: The worker must upload the completed GenAIContainer back to a specific Hugging Face repository (e.g., `pauljones0/{Original_Model_Name}-{SoC_Model}-QAIRT`).

## 2. Non-Functional Requirements  (NFRs)

### 2.1 Security & Abuse Prevention
- **NFR1**: All users must be authenticated before submitting a job (Google/GitHub via Firebase). The backend API must explicitly validate the Firebase JWT token to prevent queue-flooding from synthetic requests, extracting the secure UID natively.
  - **NFR1.1**: The system must sanitize all inbound URL payloads and enforce a global queue threshold (max 50 jobs) to neutralize queue-flooding DDoS attempts.
- **NFR2**: The system must enforce a global dynamic active billing limit (e.g. $1.00/hr burn rate) capping the pool of active worker Spot VMs at any time to prevent runaway cloud billing, thus allowing horizontal scaling without financial risk. This must be safeguarded against race conditions using atomic database transactions.
- **NFR3**: The user's specific reason for a size-based or file-size rejection must be clearly communicated. The system must inform the user of the recommended maximum size for their requested hardware and provide actionable insights (e.g., suggesting a smaller parameter size) to ensure a high-quality UX.
- **NFR4**: The Hugging Face Write Token must never be exposed to the frontend or the user. It must be injected securely into the worker VM via GCP Secret Manager at runtime (via Cloud Batch Environment Secrets natively, avoiding manual scripts).
- **NFR5**: Compute worker instances must not have external Public IP addresses. Outbound traffic to the Hugging Face hub must traverse a secure Cloud NAT on a Custom VPC.
- **NFR6**: Internal webhooks (like the worker's status update ping) must enforce strict Google Cloud IAM Identity validation rather than relying on assumed/hardcoded bearer secrets. Worker Service Accounts must explicitly lack database mutation privileges to enforce least privilege.
- **NFR6.1**: The worker node must execute pre-compilation ClamAV signature scanning on payload data and encapsulate compilation execution completely inside a user-space kernel sandbox (gVisor/runsc) with absolutely no network access (`--network none`). The Hugging Face tokens must be kept strictly on the host VM outside the sandbox.

### 2.2 Performance & Reliability
- **NFR7**: The UI must stream real-time job tracking via `onSnapshot` without HTTP polling timeouts, and dynamic waiting room ETAs must update instantaneously as the queue clears. Animations must maintain a smooth 60fps frame rate for premium aesthetic fidelity.
- **NFR8**: Worker node compilation scripts must exit cleanly (0 for success, non-zero for failure) to allow Google Cloud Batch to natively handle retries or final job failure states reliably without ghost instances.
- **NFR9**: A backend reconciliation chron job ("Reaper") must identify and safely terminate "Ghost Jobs" (instances stuck in a compiling state due to unrecoverable OS kernel panics or total node failure) to prevent single failures from permanently locking the queue.
- **NFR10**: Worker nodes must boot consistently in under 60 seconds (utilizing HashiCorp Packer custom images rather than raw environment bootstrapping). The worker Service Account (`hexforge-worker-sa`) must explicitly enforce the principle of least privilege. It must NOT have database access, and it must only have `roles/storage.objectUser` limits (not `objectAdmin`) on the Checkpoint GCS bucket to strictly prohibit ACL mutation. It must only possess `roles/secretmanager.secretAccessor` scoped strictly to the Hugging Face token secret alone. Outbound connections must funnel through a Cloud NAT attached to a Private VPC.
- **NFR11**: Extensive memory overflow during compilation must be absorbed by Local SSD NVMe drives instead of standard persistent disks to prevent IOPS bottlenecking. Local SSDs must be partitioned using reliable `dd` swapfile generation to prevent fragmentation. The `dd` procedure must use a memory-efficient `bs=1M` block size instead of 1G to prevent host memory exhaustion during block zeroing on small Spot VMs.
- **NFR12**: Worker execution must be resilient to Spot VM preemption. Expensive external operations (multi-gigabyte downloads) and multi-hour compilations must be checkpointed to an internal GCS bucket, allowing a substituted worker to resume exactly where the preempted worker halted.
- **NFR13**: The system must enforce End-to-End Structured Logging. Every job must be assigned a unique UUID Trace ID at the frontend client. This `TRACE_ID` must propagate through the HTTP boundaries, Cloud Tasks payload, and into the Google Cloud Batch worker environment variables. Each handler invocation must generate a unique **Span ID** (16-char hex) for per-invocation log grouping, and worker scripts must generate per-phase span IDs (download, compile, upload). All worker scripts **and Go Cloud Functions** must emit standard output as GCP-compliant JSON payloads containing this Trace ID (with `severity`, `message`, `component`, `timestamp` (ISO-8601), `logging.googleapis.com/trace`, `logging.googleapis.com/spanId`, `logging.googleapis.com/insertId` (for log deduplication), and `logging.googleapis.com/labels` (containing `job_id` and `user_id` where applicable) fields) to enable unified log correlation in Google Cloud Logging. Every runtime (Go, Bash, Python) must include a `component` tag (e.g. `"SubmitJob"`, `"BatchEntrypoint"`, `"CompileEngine"`) to enable per-component log filtering. The Go backend must parse `X-Cloud-Trace-Context` headers for GCP native trace span linking. Worker scripts must record phase-level timing (download, compile, upload durations) for performance telemetry. The `trace_id` must also be persisted in the Firestore `Jobs` document for database-level traceability.
- **NFR14**: The system must implement a Dead-Letter Queue (DLQ) handler. The primary Cloud Tasks queue must be configured with a Terraform-managed `dead_letter_config` routing permanently failed tasks to a secondary DLQ queue (`hexforge-dlq`). A dedicated Cloud Function must catch tasks from the DLQ, mark the corresponding Firestore job as `Failed`, release any FinOps budget lock held by the job, publish a `job.dead_lettered` event to Pub/Sub (with explicitly populated `Trace ID` and `Span ID` correlation data), and emit a `CRITICAL` severity structured log for operational alerting. The StaleJobReaper must also publish `job.reaped` events to Pub/Sub for reaped ghost jobs.
- **NFR15**: The system must enforce idempotent Cloud Task dispatch. Task names must be deterministically derived from the `job_id` (format: `hexforge-{job_id}`) to guarantee that retried `SubmitJob` calls never create duplicate tasks in the Cloud Tasks queue.
- NFR16: The system must implement a Pub/Sub lifecycle event bus (`hexforge-job-events`). Job lifecycle events (`job.submitted`, `job.provisioning`, `job.downloading`, `job.compiling`, `job.uploading`, `job.completed`, `job.failed`, `job.dead_lettered`, `job.reaped`) must be published with message ordering by `job_id` and 7-day message retention. Events must be published by ALL pipeline stages: Go Cloud Functions for infrastructure states and Worker scripts for compilation phases. A push subscription must route events to an `EventProcessor` Cloud Function for centralized audit logging and high-priority alerting on primary lifecycle failures. A secondary pull subscription (`hexforge-job-events-sub`) must be provisioned for general-purpose external consumers. Publishing must use `context.Background()` with a timeout to prevent silent failures from cancelled request contexts. Publishing must be fire-and-forget: failures must be logged but must never block the main pipeline. All FinOps cost estimation constants must be centralized in a shared module to prevent coefficient drift across handlers.
- **NFR17**: The system must enforce per-handler Span ID propagation. Each Go Cloud Function handler must generate a unique 16-character hex Span ID at entry and include it in all log entries and Pub/Sub events. Worker scripts must generate per-phase span IDs (download, compile, upload) and propagate the compile-phase span ID into the gVisor sandbox via the `SPAN_ID` environment variable.
- NFR18: The system must enforce log deduplication via `insertId`. Every structured log entry across all runtimes (Go, Bash, Python) must include a `logging.googleapis.com/insertId` field — UUIDs in Go, sequential counters in worker scripts — to enable GCP Cloud Logging to collapse duplicate entries from retried Cloud Function invocations or Cloud Batch task restarts.
- NFR19: The system must enforce a centralized event type catalog. All event names must be defined as typed constants in a single module (`back_end/handler/events.go`) and validated by the `EventProcessor` to prevent architectural drift and ensure schema enforcement across the entire distributed pipeline.
- **NFR20**: The backend must have a comprehensive automated unit-test suite covering: the `SubmitJobUseCase` (including happy path, cache hits, queue-full, per-user quota enforcement, enqueue failures, and shell-injection rejection), `EstimateJobCost` FinOps calculations, and all hardware profile and device dictionary mappings. Business logic must be tested in isolation using in-memory mocks for all external dependencies (Firestore, Pub/Sub, Cloud Tasks). The absence of an event publication when an enqueue fails must be explicitly asserted.
- **NFR21**: A CI pipeline (GitHub Actions) must run `go vet ./...`, `go build ./...`, and `go test ./... -race` on every push or pull request to the `main` branch that modifies `back_end/**` files. The race detector must be enabled to catch concurrency bugs in shared mock state and handler logic.
