# HexForge Design Document

## 1. User Interface & User Experience (UI/UX)
**Theme**: "Sci-Fi / Semiconductor"
- **Aesthetic**: Premium Dark Mode, glassmorphism (`backdrop-blur`), subtle emerald glowing accents (`box-shadow` neon glows), and highly responsive micro-animations.
- **Framework**: React / Next.js 14.
- **Styling**: Tailwind CSS for core styling, complemented by Lucide Icons for visual anchors.
- **Experience Flow**:
  1. User authenticates via Google/GitHub (to prevent anonymous abuse).
  2. Selects their Snapdragon device from a dynamically populated dropdown list fetched from the backend database on initial page load.
  3. Provides a Hugging Face GGUF URL.
  4. The UI performs instant frontend validation, parsing the URL's parameter size (including decimals like 1.5B via robust float parsing) against the selected phone's limit. If the model is too large, the "Init Sequence" button is disabled and an actionable warning is rendered (e.g., "Max limit is 3B parameters for this device. Please select a smaller model.").
  5. Clicks "Init Sequence".
  6. The backend performs a secondary validation check and an actual file-size verification against the Hugging Face API to prevent bypassing limits.
  7. If the requested output model already exists in the destination Hugging Face repository, the backend instantly completes the job (Cache Hit), bypassing the worker queue entirely.
  8. Upon successful validation and queue insertion, the UI transitions to a real-time "Pipeline Telemetry" tracker showing exact states (Queued -> Provisioning -> Compiling -> Uploading) streamed via Firestore `onSnapshot`. This streaming logic, alongside Auth and Hardware Profiles, is strictly abstracted into isolated custom hooks (`useJobTelemetry`, etc.) to maintain a clean Presentation/Data separation in the Next.js `app` router.
  9. While in the "Queued" state, the UI orchestrates a "waiting room" experience, dynamically displaying the user's real-time queue position and an Estimated Time of Arrival (ETA) to reduce page abandonment psychology.
  10. If the user closes the browser during tracking, the backend compilation proceeds safely in the background.

## 2. System Architecture Design
The system employs a heavily decoupled, asynchronous design to isolate the cheap static frontend from the expensive, volatile compilation backend.

### 2.1 The Orchestrator (Serverless)
- **Technology**: Golang, Google Cloud Functions, Cloud Tasks, Firestore (NoSQL).
- **Design Choice**: Cloud Functions are used as an API gateway because they scale to zero and cost fractions of a cent. To prevent technical debt and ensure long-term maintainability, the Golang codebase is structured using **Domain-Driven/Hexagonal Architecture**. The monolithic HTTP handlers are decoupled into clean `domain` ports (interfaces like `JobRepository`, `EventPublisher`), concrete `infrastructure` adapters (Firestore and Pub/Sub implementations), and testable `usecase` business logic blocks. The Cloud Functions strictly act as the transport layer invoking these usecases. Golang provides type-safe mapping of hardware profiles and lightning-fast execution. The `UpdateJobStatus` function strictly requires IAM Cloud Identity authentication, blocking anonymous or fabricated worker status updates. To prevent billing runaway and function starvation, Cloud Functions are hard-capped to a maximum of 10 instances via Terraform.
- **Infrastructure as Code**: The GCP infrastructure is defined via **modular Terraform** split into `apis`, `networking`, `iam`, `storage`, `queues`, and `serverless` modules in `infra/modules/`, orchestrated by `infra/root.tf`. This eliminates the 661-line monolithic `main.tf` and achieves clean separation of concerns—each module owns a distinct GCP layer.
- **Queue Management & Database Architecture**: The system utilizes **Cloud Tasks** for robust event-driven job queueing, featuring built-in exponential backoff and dead-letter queue (DLQ) support. Cloud Tasks are created with **idempotent deterministic names** derived from the `job_id`, guaranteeing that retried `SubmitJob` calls never create duplicate tasks. Firestore serves as the central source of truth for "Queue State", bridging the HTTP frontend and the Bash worker nodes seamlessly. To prevent catastrophic queue race conditions that could bypass the billing limits, the system utilizes a singleton document `System/ConcurrencyLock` to track `active_spend_rate_cents_per_hr`. When the `QueueManager` task executes, a `firestoreClient.RunTransaction()` block explicitly reads the current active burn rate, aggressively coercing it to `float64` to prevent panic or budget calculation drift. If the requested node's cost strictly stays under the dynamic FinOps budget cap (e.g., max $1.00/hr active spend), it atomically increments the lock and provisions the Batch VM. Cost estimation is granular: each architecture (v68, v73, v75, v79) is assigned a specific vCPU count (8-32 cores) and RAM footprint, ensuring highly accurate budget enforcement. **Crucially**, if the budget cap is exceeded, the `QueueManager` returns an HTTP 429 status code. Cloud Tasks intercepts this and automatically invokes its exponential backoff logic, retrying the task safely later without requiring custom polling loops. All FinOps cost estimation constants and job status event types are centralized in shared modules (`finops.go`, `events.go`) to prevent coefficient drift and architectural divergence across handlers.
- **End-to-End Telemetry**: Every compilation job originates at the frontend with a unique UUID `Trace ID`. This ID propagates downward through the HTTP headers (to `SubmitJob`), into the Cloud Tasks payload, as an Environment Variable into the Batch Worker VM, and finally binds to all JSON-formatted **GCP Cloud Logging** stdout emissions. Each handler invocation generates a unique **Span ID** (16-char hex via `crypto/rand`) for per-invocation log grouping in Cloud Trace. High-latency external calls, such as the Hugging Face HEAD request, are wrapped in explicit sub-spans (`spanID_hf`) for granular waterfall profiling. Every log entry across all three runtimes (Go Cloud Functions, Bash entrypoint, Python compiler) includes: a `component` tag (e.g. `"SubmitJob"`, `"BatchEntrypoint"`, `"CompileEngine"`), an ISO-8601 `timestamp`, structured `logging.googleapis.com/labels` (containing `job_id`, `user_id` where applicable), `logging.googleapis.com/spanId` for per-handler/phase trace grouping, and `logging.googleapis.com/insertId` (UUID in Go, sequential counter in workers) for log deduplication on retried invocations. The Go backend also parses `X-Cloud-Trace-Context` headers for GCP native span linking. Worker scripts emit phase-level performance timing (download, compile, upload durations) for operational insights.
- **Dead-Letter Queue (DLQ)**: The primary Cloud Tasks queue (`hexforge-job-queue`) is configured with a Terraform-managed `dead_letter_config` that routes permanently failed tasks to a secondary queue (`hexforge-dlq`). When a task lands in the DLQ queue, it is dispatched to the `DeadLetterHandler` Cloud Function. This handler marks the Firestore job as `Failed`, decrements the `ConcurrencyLock` spend rate if the job held an active lock, publishes a `job.dead_lettered` event to Pub/Sub, and emits a `CRITICAL` structured log. This prevents silent task disappearance from permanently blocking the job queue.
- **Pub/Sub Event Bus**: Job lifecycle events (`job.submitted`, `job.provisioning`, `job.downloading`, `job.compiling`, `job.uploading`, `job.completed`, `job.failed`, `job.dead_lettered`, `job.reaped`) are published to a `hexforge-job-events` Pub/Sub topic with 7-day message retention and message ordering by `job_id`. Events are published by Go Cloud Functions and directly from Worker nodes using `gcloud pubsub`. A push subscription (`hexforge-job-events-push`) routes these events to an **EventProcessor** Cloud Function, which performs unified audit logging and routes critical states (failures, reaps, dead letters) to elevated alerting severity. A legacy pull subscription (`hexforge-job-events-sub`) is also provisioned with a dead-letter policy (max 5 delivery attempts → `hexforge-job-events-dlq` topic) for external consumers. This enables downstream consumers (Slack alerts, analytics dashboards, mobile push notifications) without modifying the core pipeline. Publishing uses `context.Background()` with a 5-second timeout to prevent silent failures, and is fire-and-forget: failures are logged but never block the main pipeline.
- **Ghost Job Monitor / Reaper & Alerting**: To handle hard-crashes on the worker compute plane (e.g. OOM lockups before error statuses can be emitted), a scheduled Cloud Function acts as a "Reaper". It queries Firestore for any job stuck in active states (`Provisioning`, `Downloading`, `Compiling`, etc.) for more than 4 hours. It automatically transitions the job to `Failed`, explicitly decrements the `ConcurrencyLock`'s `active_spend_rate_cents_per_hr` using a Firestore Transaction for atomicity, and publishes a `job.reaped` event to Pub/Sub to unblock the main queue. All Cloud Tasks dispatches are secured with OIDC authentication tokens, ensuring only verified service accounts can invoke the `QueueManager` endpoint. Furthermore, the infrastructure is equipped with a Google Cloud Monitoring Alert Policy (`hexforge/reaped_jobs_count`) that triggers an immediate incident response to the SRE team whenever a Ghost Job is detected.

### 2.2 The Compute Plane (Google Cloud Batch)
- **Technology**: Google Cloud Batch, Compute Engine, HashiCorp Packer, Bash, Python, Private VPC.
- **Design Choice (Cloud Batch & Networking)**: Rather than manually managing a pool of VMs and writing resilient polling loops, the system entrusts job execution to Google Cloud Batch. The workload is strictly deployed into a Custom VPC using Private Google Access and Cloud NAT, guaranteeing that compiling nodes have NO external public IPs, drastically reducing horizontal attack vectors. The `QueueManager` simply submits a Job definition, and GCP autonomously provisions the VM, runs the compiler, handles retries on preemption, and terminates the node without any "TTL" guesswork or custom self-destruct logic.
- **Design Choice (Spot Instances & Checkpointing)**: To slash standard compute costs by 60-80%, Google Cloud Batch targets **Spot VMs**. Because Spot Instances can execute without warning, the `batch_entrypoint.sh` implements an intermediate GCS Checkpoint bucket mechanism. If a VM is preempted during a multi-hour compile, the subsequent VM retrieves the previously downloaded model and compiled state from GCS instead of repeating the entire pipeline phases.
- **Design Choice (Custom Machine Types & Local SSDs)**: The workflow dynamically defines Google Cloud Batch tasks using Custom Machine Types based precisely on the job's RAM requirements. Furthermore, Local SSDs (ephemeral NVMe drives) are natively attached and formatted as high-speed Swap space to prevent disk-IO bottlenecking during intense OOM serialization spikes.
- **Design Choice (Packer Base Image)**: Installing the massive QAIRT SDK and dependencies on-the-fly takes minutes. Packer pre-bakes an Ubuntu 24.04 LTS snapshot, reducing worker boot-up to ~30 seconds when Cloud Batch spawns it.
- **Design Choice (Hardware-Aware Graph Optimization)**: To prevent execution fallback to the CPU and unlock peak TOPS, the compiler natively utilizes: translates hierarchical K-Quants with `Qnn_FloatBlockEncoding_t`, enforces `BFloat16` for unstable node projections, injects `-O3` with `--dump_lut`, enforces NHWC layouts, sets `REPACK_FOR_HVX=1`, offloads logits to CPU on 32-bit cDSP, applies `--mmap_budget 25` for paging alongside **Byte-Granularity Allocation** to prevent fragmentation, enforces 64-element HMX alignment via automated padding, upcasts INT4 to Q8_0 for legacy silicon, and implements **IQ4_NL + Q8_0 Hybrid Quantization** to prioritize transformer blocks while protecting sensitive layers. It also enables **L2 Cache Elevation** to lock VTCM boundaries, **8-Bit KV Cache Compression** to slice footprint metrics in half, **Speculative Decoding (SSD-Q1, Eaglet, LADE)** to reclaim idle HMX compute rows, **RoPE AoS Interleaving** (with AOT interleaving pass for SoA source files), **Profile Bucketing** (multi-binary generation), and **Advanced Graph Fusion** (SwiGLU/FMHA). For v80/v79, it leverages **Mixed-Precision FP8 (E4M3/E5M2)** and Windows-on-ARM threading hints.
- **Design Choice (Macro-Level Connections)**: Automatically derives attention head dimensionality ($E_{total} / N_{heads}$) when missing and parses `architecture.connector` to map native sequential or parallel topologies precisely.
- **Design Choice (LiteRT Abstraction)**: To prevent strict QNN constraint failures on highly unorthodox graphs, the compiler invokes a JIT fallback loop targeting the LiteRT Unified API.
- **Design Choice (Sandboxing, Airgapping & Security)**: To prevent malware compilation, data exfiltration, or host machine escapes, the Python compilation process is encapsulated within a Docker container utilizing the `runsc` (gVisor) runtime and `--network none`. The sandbox is hardened with `--security-opt=no-new-privileges` and a restricted `tmpfs` for internal scratch space, ensuring no accidental privilege Escalation or persistence. The Hugging Face token is strictly kept outside the sandbox. This provides defense-in-depth kernel and network isolation. Furthermore, a ClamAV scan phase asserts payload safety before compilation is allowed to execute. The API layer also enforces a repository blacklist to proactively block known malicious sources.
- **Design Choice (Deployment Standalone Architecture)**: To eliminate secondary runtime dependencies, the compiler produces a **standalone execution bundle** including the compiled context binaries, the Hugging Face Fast Tokenizer JSON exported via `--export_tokenizer_json`, and any required LoRA adapters with `skip-lora-validation` set for rapid user-facing startup.
- **Design Choice (MoE Graph Switching)**: Supported on HTP Generation 3+ devices to optimize Mixture of Experts inference by partitioning experts into sequential subgraphs or redundant memory segments.
- **Design Choice (KV Cache Spill-Fill)**: Implements a dedicated spill-fill buffer to seamlessly page KV cache fragments from VTCM out to LPDDR memory, preventing allocator crashes during ultra-long context generation.
- **Design Choice (Universal Custom Op Fallback)**: Automatically invokes the OpPackage generator skeleton to bridge unsupported mathematical nodes (e.g., HardSwish, Mish, SwiGLU) via DSP-native C++ math kernels. The generator enforces **BHWC 4D layout** and `const` parameter constraints (via `setReadOnly(true)`) and ensures that every LOAD is **128-byte aligned** for HVX vector registers.
- **Design Choice (Explicit Tensor Padding)**: Performs an execution safety pass injecting `--padding.baseline 128` across the trailing dimensions of all non-HMX tensors, aligning memory access shapes to the exact width of the HVX vector processing hardware.
- **Design Choice (v79/v80 Extended Precision)**: Leverages native INT2 and FP8 support on Hexagon v79 and v80 to achieve higher throughput for optimized model variants.
- **Design Choice (LongRoPE Scaling)**: Supports rotary scaling for extended context windows by injecting `--longrope` and specific frequency factors from GGUF metadata.
- **Design Choice (Calibration & Layout Enforcement)**: Enforces Crouton layout and VTCM residency via `tensor_properties.json` and supports representative calibration dataset ingestion to optimize fixed-point scaling factors. It also dynamically configures normalization (RMS-norm) and activation (gated SwiGLU) parameters based on GGUF metadata.
- **Design Choice (SpinQuant Orthogonal Rotations)**: Mitigates performance loss inherently tied to affine block quantization by orthogonally rotating mathematically extreme weights into safer margins before 4-bit rounding occurs.
- **Design Choice (Sequential MSE Scale Refinement)**: Replaces traditional min/max calibration ranges with an intensive offline iterative search, discovering precise scale combinations that minimize the mathematical error against FP16 baselines.
- **Design Choice (Mathematical FP32 Firebreaks)**: Uses `--volatile_node_precision FP32` to isolate highly-volatile summation blocks, halting the uncontrolled propagation of Infinity and NaN values that cause generative degradation.
- **Design Choice (QNNHardwareBridge Interception Layer)**: Before compilation, GGUF metadata is filtered through an abstraction bridge that explicitly translates conceptual architectures (like DeepSeek or Mistral) into rigid Hexagon operations, handling edge cases such as unrolling MLA natively to dense attention maps before the QNN compiler evaluates the graph.
- **Design Choice (Advanced Graph Surgeries & Outlier Suppression)**: The compiler natively applies Olive-like graph node modifications (such as `MatmulAddFusion` incorporating exact mappings for `MaskedSoftmax` and `MatMul_bias`, `SimplifiedLayerNormToL2Norm`, `RemoveRopeMultiCache`, and `AttentionMaskToSequenceLengths`). It specifically deploys `WeightRotation` to suppress 4-bit quantization outliers prior to `tuner.optimize` VTCM tiling experiments.
- **Design Choice (Explicit Target SoC Assignment & Native FP16 Scaling)**: Replaces generic architectural definitions by injecting exact SoC models (`QNN_SOC_MODEL_SM8750`) and forces runtime graph calculations of GGUF K-Quants natively onto hardware HMX units via `QNN_PRECISION_FLOAT16` configuration.
- **Design Choice (W4A16 Dequantization)**: For hardware that usually forces Q8_0 fallback, the compiler instead maintains INT4 weights in storage and dynamically injects `fp16_kernels` to dequantize them in registers, saving critical DRAM bandwidth.
- **Design Choice (Tightly Coupled Memory Contexts & Packed Channel Limits)**: Generates robust hardware barriers by mathematically enforcing `ceil(channels / 4.0)` spatial packed ceilings. Protects host memory via strict FileReadMemoryBudget chunk limits and utilizes `QNN_HTP_MEM_SHARED_BUFFER` for zero-copy unified inference updates without power-intensive re-duplication.
- **Design Choice (HTP Compiler Graph Rewriter)**: Utilizes `HTPCompilerGraphRewriter` before translation to enforce 4D BHWC shapes and strictly zero quantization offsets, structurally preventing QNN SDK compiler panics.
- **Design Choice (Transformer Composer JSON Alignment)**: Programmatically overrides standard builder inputs with precise JSON to mandate 32-byte hardware cache alignment, Array of Structures memory formatting, and dynamic context allocation bounding.
### 2.3 Hardware Constraint Oracle
The compilation engine is informed by a rich hardware constraint model that tracks the physical capabilities and limitations of each Hexagon NPU architecture. This oracle dictates compilation decisions including graph chunking, memory layout, and operator placement.

- **VTCM (Vector Tightly Coupled Memory)**: Each Hexagon architecture provides a dedicated on-chip SRAM pool shared between the Scalar, Vector, and Tensor accelerators. Sizes range from 512KB (Hexagon 698) up to 8MB+ (v81). The compiler employs byte-granularity algorithms (`set_vtcm_size_bytes`) to specifically allocate perfectly-sized chunks rather than defaulting to megabyte tiering, maximizing this boundary. It also injects **L2 Cache Elevation** flags, dynamically coercing the Snapdragon's Level 2 Cache into acting as extended VTCM.
- **Crouton Memory Layout**: The HTP mandates a proprietary chunked memory layout (`ChunkedMemoryLayout<4, 0,0, 1,0, 2,0, 3,0, 1,8, 2,8, 3,32>`) where tensors are organized into `1×8×8×32` contiguous blocks. Tensors that don't align to these multiples are automatically padded with the value **31** (not 0). Standard `memcpy` operations cannot cross chunk boundaries. The compiler enforces Crouton layout via `DEF_TENSOR_PROPERTIES` macros.
- **HVX 128-byte Register Alignment**: The Hexagon Vector eXtension (HVX) registers are exactly 128 bytes wide. A single GGUF Q4_0 quantization group (32 weights × 4 bits = 16 bytes) occupies only 1/8th of an HVX register. The compiler coalesces 8 groups into 128-byte super-groups via `REPACK_FOR_HVX=1` to fully saturate the vector pipeline.
- **32-bit cDSP Address Space (3.75GB Limit)**: The Hexagon cDSP is a 32-bit processor, limiting each QNN session to ~3.75GB of addressable memory. The compiler calculates the estimated session memory using a refined heuristic that accounts for weight precision, KV cache geometry parameterized by bitwidth compression requests (8-bit vs 16-bit), and vocabulary projection spikes: $M_{est} = (P_{total} \times 0.5 \text{ bytes}) + (KV_{size}^{FP16 \text{ or } INT8} \times 1.3) + Logits_{spike}$. The `--mmap_budget 25` flag alongside precise **byte-granularity allocation** ensures context binaries are paged incrementally rather than demanding contiguous legacy megabyte allocation blocks, reducing internal fragmentation.
- **NHWC Layout Enforcement**: The HTP natively requires NHWC (TensorFlow-style) tensor ordering. Models originating from PyTorch (NCHW) would trigger expensive runtime Transpose ops. The compiler forces `--force_nhwc 1` to permute static weight tensors at compile time.
- **Logits Tensor Offloading**: For architectures with constrained session memory, the massive vocabulary projection tensor is offloaded to the CPU via `--lm_head_backend CPU` to prevent address space overflow, trading slight latency for system stability.
- **HMX Head-Dimension Alignment**: The HTP-HMX unit mandates that attention head dimensions be multiples of 64. The compiler forcefully injects zero-padding for non-compliant models to maintain hardware acceleration and numerical stability. **Perplexity hemorrhaging** is proactively avoided by ensuring the hardware only interacts with perfectly divisible tensor dimensions.
- **GQA / MQA Broadcasting**: To support modern transformer topologies (Llama 3, Mistral) on the HTP, the compiler extracts `num_kv_heads` and injects native broadcast logic to map Grouped-Query Attention efficiently.
- **Legacy INT4 Safety Fallback**: For v68/v69 silicon lacking native INT4, the compiler upcasts weights to Q8_0 at build time to prevent severe software dequantization performance degradation.
- **Speculative Decoding (SSD-Q1, Eaglet, LADE)**: The compiler reclamation logic injects speculative decoding branches into the Genie Dialog JSON. This transforms memory-bound decoding (GEMV) back into compute-saturated matrix multiplications (GEMM), utilizing the 31 idle rows of HMX hardware per clock cycle.
- **RoPE AoS Interleaving**: While GGUF uses SoA for complex numbers, the HTP achieves peak vector MAC efficiency with Array of Structures (AoS). The compiler forces AoS interleaving (via `repo_complex_organization` and `tensor.kq_complex_organization`) during weight generation.
- **Multi-Session Shared Memory**: When splitting models to bypass the 3.75GB FastRPC wall, the compiler enables `in-memory-kv-share` and **Q8_0_32 KV Quantization**, allowing the conversation history to persist across session boundaries in shared LPDDR space.

### 2.4 HTP Silicon-Level Optimization Principles

#### 2.4.1 Crouton Memory Layout
To minimize DDR stalls, the compiler enforces the **Crouton layout**. This physically reorganizes linear tensors into blocked, Z-order spatial tiles (typically 1x8x8x32). This layout ensures that a 256-byte vector load fetches only relevant, contiguous data for matrix multiplications.

#### 2.4.2 HVX Register Saturation
Standard 4-bit quantization groups are too narrow for the 128-byte HVX registers. The compiler uses hierarchical re-packing (`REPACK_FOR_HVX=1`) to coalesce groups, preventing 87.5% bandwidth waste during vector math.

#### 2.4.3 Test-Time Scaling (Speculative Decoding)
The compiler transforms decoding (GEMV) into compute-saturated GEMM by injecting speculative parallel decoding branches. This utilizes the otherwise idle rows in the 32x32 HMX matrix grids.

## 3. Data Flow
1. **Frontend** -> (HTTP POST w/ `X-Trace-ID`) -> **Cloud Function API (`SubmitJob`)**
2. **Cloud Function API** -> (Validates JWT Token & Writes) -> **Firestore (`status: Queued`)**
3. **Cloud Function API** -> (Dispatches Idempotent Task) -> **Cloud Tasks Queue**
4. **Cloud Tasks Queue** -> (Invokes with Backoff/Retries) -> **Cloud Function API (`QueueManager`)**
5. **Cloud Function API** -> (Submits Job w/ `TRACE_ID` Env) -> **Google Cloud Batch**
6. **Cloud Batch Managed VM** -> (Provisioned with Local SSD)
7. **Worker VM Bash Entrypoint** -> (Updates Status w/ Trace ID, Logs JSON Traces with Timestamps, Span IDs, InsertIds & Labels) -> **Firestore / Cloud Logging**
8. **Worker VM Bash Entrypoint / Host** -> (Downloads Payload) -> **Hugging Face Hub**
9. **Worker VM Sandbox** -> (Airgapped gVisor Compilation, Component-Tagged Logs) -> **Host Output Dir / Cloud Logging**
10. **Worker VM Bash Entrypoint / Host** -> (Uploads Artifact) -> **Hugging Face Hub**
11. **Cloud Tasks DLQ** -> (Max Retries Exhausted, Routes to `hexforge-dlq`) -> **DeadLetterHandler** -> (Marks Failed, Releases Lock) -> **Firestore**
12. **All Handlers (Functions & Worker)** -> (Lifecycle Events: submitted/provisioning/downloading/compiling/uploading/completed/failed/dead_lettered/reaped) -> **Pub/Sub (`hexforge-job-events`)**
13. **Pub/Sub Event Bus** -> (Push Subscription) -> **EventProcessor Cloud Function** -> (Audit Logging & Critical Alerting) -> **Cloud Logging**
14. **Frontend** -> (Streams via Firestore `onSnapshot` real-time listeners) -> **Firestore**
